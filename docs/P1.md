# P1 — Per-roof solar yield (physics-first) + the first yield map

This is the second build phase. In one sentence:

> **P1 turns the 3D shapes of buildings into an estimate of how much solar
> electricity each individual roof surface could generate in a year, and paints
> that onto an interactive map.**

It uses *physics*, not guesswork or AI — the same kind of calculation the official
PVGIS tool uses — and then checks its own answer against PVGIS to prove it's right.

---

## 1. What you need first (prerequisites)

P1 reads the data that **P0 already downloaded**. So before P1 you must have run
P0 at least once for your area:

```bash
rsgt ingest -c configs/oudewater.yaml      # P0 — downloads the data (do this first)
```

You also need the solar maths library (`pvlib`). Install the project with the
`geo` and `solar` extras:

```bash
pip install -e ".[geo,solar]"
```

That's it. Everything else is automatic.

---

## 2. The one command

```bash
rsgt yield -c configs/example_small.yaml -v
```

(Use `configs/oudewater.yaml` for the whole town; `example_small.yaml` is a tiny
test block that finishes in well under a minute.)

When it finishes you get, under `data/processed/`:

| File | What it is |
|---|---|
| **`<aoi>_yield_map.html`** | **Open this in a browser.** An interactive map; every roof coloured by how good it is for solar. |
| `<aoi>_roof_yield.gpkg` | The full table: one row per roof surface, with its size, angle, direction and yield. |
| `<aoi>_building_yield.csv` | A simpler per-building summary (one row per building). |
| `<aoi>_roofs.geojson` | The same roof data as plain GeoJSON (for other tools). |
| `<aoi>_yield_summary.json` | A short machine-readable report of the whole run. |

**To see the result, just double-click `<aoi>_yield_map.html`.** It's a single
self-contained file — no server needed (you only need internet for the background
map tiles).

## 3. What happens inside, in plain language

Running `rsgt yield` performs six steps. You don't have to run them separately,
but here is what each one does and why.

### Step 1 — Find every roof surface ("roof planes")
The 3D BAG data describes each building as a little 3D model. A house with a
classic pitched roof has (at least) two roof surfaces: one facing one way, one
facing the other. We read each building and pull out its roof surfaces, and for
each one we measure three things straight from its 3D shape:

- **Area** — how big the roof surface is, in square metres.
- **Tilt** — how steep it is. `0°` = flat, `90°` = a vertical wall. A typical
  Dutch pitched roof is around `30–40°`.
- **Azimuth** — which compass direction it faces. `180°` = due south (the best in
  the Netherlands), `90°` = east, `270°` = west, `0°/360°` = north (the worst).

We then throw away surfaces that aren't worth solar panels: ones that are too
small, or both steep **and** north-facing.

> Run just this step on its own with `rsgt roofs -c <config>` if you only want the
> roof geometry without the solar maths.

### Step 2 — Get the local sunshine ("solar resource")
We ask **PVGIS** (the European Commission's free solar database) for a *Typical
Meteorological Year* (TMY) at the centre of your area: 8 760 hourly values (one per
hour of a representative year) of how much sunlight hits the ground, plus air
temperature and wind. This is downloaded once and cached, so it's instant next time.

### Step 3 — Work out the sunshine *on each roof* and turn it into electricity
Sunlight hitting a *tilted, south-facing* roof is more than sunlight hitting flat
ground — and a north roof gets much less. For every roof we re-project the hourly
sunshine onto that roof's specific tilt and direction (this is called
*plane-of-array* irradiance), then run a standard solar-panel model (**PVWatts**)
that accounts for:

- panel temperature (hot panels are slightly less efficient),
- normal system losses (wiring, dirt, etc. — about 14%),
- the inverter (and its "clipping" on the sunniest moments).

The result for each roof is two numbers:

- **Specific yield** (`kWh/kWp` per year) — the *quality* of the roof: how many
  units of electricity each unit of panel capacity makes per year. South ~1050,
  north ~600 in the Netherlands.
- **Annual energy** (`kWh` per year) — the *total*, = quality × how many panels
  fit (which depends on the roof's size).

### Step 4 — Add context (who/where each roof is)
We attach each roof's **construction year** and **use** (from BAG) and its
**6-digit postcode / PC6** (the unit the grid-congestion data uses later). We also
roll the per-roof numbers up to a **per-building** summary.

### Step 5 — Draw the map
We export the roofs to a self-contained **deck.gl** web map, coloured by specific
yield. Hover any roof to see its full breakdown. This is the "early visible win"
of the project.

### Step 6 — Check our answer against PVGIS ("validation")
Finally, for a standard reference roof (south-facing, 35° tilt) we compare *our*
yield against *PVGIS's own* published estimate for the same spot. If the two agree
closely, our physics is trustworthy.

## 4. A worked example (the Oudewater test block)

Running `rsgt yield -c configs/example_small.yaml` on the sample block gives:

- **1 026 usable roof planes** across **512 buildings**
- **3 341 kWp** of installable solar capacity, **≈ 2.9 GWh/year** of technical potential
- **Median specific yield 909 kWh/kWp** (a mix of all roof directions)
- South-facing roofs ≈ **1 042 kWh/kWp** vs north-facing ≈ **612** — exactly the
  physics you'd expect (~1.7× better).
- **Validation: ours = 1 054 vs PVGIS = 1 039 kWh/kWp → +1.4% difference.**
  That tiny gap is the headline confidence check: our bottom-up physics reproduces
  the official tool almost exactly on an unshaded south roof.

---

## 5. Understanding the numbers (a mini-glossary)

| Term | Plain meaning |
|---|---|
| **kWp** ("kilowatt-peak") | The *size* of a solar installation — its power on a perfect sunny moment. ~3–4 kWp is a typical house. |
| **kWh** | A *unit of energy*. Your electricity bill is in kWh. |
| **Specific yield (kWh/kWp)** | How many kWh one kWp makes in a year here. The roof-*quality* score. |
| **Tilt / azimuth** | Roof steepness / compass direction it faces. |
| **TMY** | "Typical Meteorological Year" — a representative year of hourly weather. |
| **POA** | "Plane-of-array" — the sunlight actually landing on the tilted roof. |

## 6. Settings you can change (all optional)

Defaults are sensible; you only touch these to experiment. Add a `solar:` or
`roof:` block to your config YAML. The defaults are:

```yaml
roof:
  min_area_m2: 5.0          # ignore roof surfaces smaller than this
  max_tilt_deg: 85.0        # ignore near-vertical surfaces
  drop_north_steep: true    # drop steep north-facing roofs (bad for solar)

solar:
  usable_fraction: 0.7              # share of a roof you can actually cover with panels
  module_power_density_kwp_m2: 0.18 # how much panel power fits per usable m^2
  system_loss_fraction: 0.14        # standard 14% system losses
  dc_ac_ratio: 1.1                  # inverter sizing
  transposition_model: haydavies    # sky model for tilted-surface sunlight
```

So a 100 m² south roof → 70 m² usable → **12.6 kWp** of panels → at ~1 050 kWh/kWp
→ **~13 200 kWh/year**.

## 7. Honesty: what P1 does *not* do yet

P1 is the **unshaded baseline** — deliberately. Being clear about this is part of
the project's integrity:

- **No shading yet.** A roof shaded by a tall neighbour or a tree is currently
  treated as if it had clear sky. Real shading is added in **P2** (using the AHN
  height data P0 already downloaded). This is *why* the per-roof number is an
  upper bound and where the later machine-learning correction earns its place.
- **Flat roofs** are modelled lying flat (a conservative, lower estimate); in
  reality panels are usually tilted up on frames.
- **One weather year for the whole area.** Fine for a single small municipality;
  the sunshine barely changes across a few kilometres.

## 8. Command cheat-sheet

```bash
# the whole P1 pipeline (roofs + physics + map + validation)
rsgt yield -c configs/oudewater.yaml -v

# only extract roof planes (step 1), no solar maths
rsgt roofs -c configs/oudewater.yaml

# recompute from scratch, ignoring caches
rsgt yield -c configs/oudewater.yaml --force

# skip the PVGIS validation (so it runs fully offline once the TMY is cached)
rsgt yield -c configs/oudewater.yaml --no-validation

# skip building the map
rsgt yield -c configs/oudewater.yaml --no-map
```

(`python -m rsgt yield ...` works identically.)

## 9. What's next (P2)

**P2 — shading.** Build a surface model from the AHN LiDAR height data and work
out, for each roof, how much sunlight is blocked by surrounding buildings and
trees through the day and year. That turns today's clear-sky upper bound into a
realistic figure — and is the first place the project's machine-learning angle
(a fast "shading surrogate", then a physics-residual correction) comes in.
